===============================================================================
The FAT-Engine itself actually consists of the following files:
===============================================================================

fat.c          ...... main file (contains core raytracing stuff)
scale_it.c     ...... highlevel vertical strip drawing stuff
tables.c       ...... lookup tables for sin/cos etc. (automatically generated)
scalers.c      ...... ASM code of vertical wall strip renderers (automat. gen.)
transscalers.c ...... ASM code of vertical sprite strip renderers (automat. gen.)
input.c        ...... lowlevel keyboard reading stuff
load.c         ...... FAT standard texture format files handling
fastsqrt.c     ...... fast integer square root function
unpack.c       ...... exepack decompression function
console.c      ...... debug console (for console mode version fatlib_c.dll)
logo.c         ...... FAT-Engine's splashscreen display function


===============================================================================
Additionally used Tools
===============================================================================

tools\tablemaker.c      .... generates tables.c (lookup tables)
tools\genscalers.c      .... generates scalers.c (vert. wall strip renderers)
tools\gentransscalers.c .... generates transscalers.c (vert. sprite strip renderers)


===============================================================================
About the vertical strip renderers
===============================================================================

For speedup reasons the FAT-Engine holds one renderer for each used height.
For example: there is a renderer for strip height 6, a renderer for strip
height 8 and so on.

Each vertical strip is rendered in two passes. First the upper part of a
strip is rendered and then the lower part using the same renderer.

The following code is from scalers.c and is a renderer for height 6:

-------------------------------------------------------------------------------
input parameters:
-------------------------------------------------------------------------------

%d0.l .... holds first 32-bit of texture during first call and second 32-bit
           of texture during second call
%d1.w .... mask used to set a pixel at destination address (OR value)
%d2.w .... mask used to delete a pixel at destination address (AND value)
%a0   .... points to upper part of word destination array of height 96 during
           first call and to the lower part of word destination array during
           second call.

The destination is organized as vertical strips of width 16 bits (1 word) and
height 96. This way simple word operations can be used while stepping to the
next destination word.
Wherever possible "tst.l %d0" is used instead of "btst #nr,%d0".
The numbers used for the "btst" operation are hardcoded for each renderer to
gain as much speed as possible (but they will waste of course much time this
way).
The renderers are automatically generated by tool genscalers.c (more
documentation can be found in the source code of this tool).

__scale006:
tst.l %d0
bpl.s ___a0003
or.w %d1,(%a0)+
bra.s ___c0003
___a0003:
and.w %d2,(%a0)+
___c0003:
btst #19,%d0
beq.s ___a0004
or.w %d1,(%a0)+
bra.s ___c0004
___a0004:
and.w %d2,(%a0)+
___c0004:
btst #6,%d0
beq.s ___a0005
or.w %d1,(%a0)+
rts
___a0005:
and.w %d2,(%a0)+
rts

The renderers for sprites (transscalers.c) are quite similar. The only addition
is the using of another register which holds the transparency mask and which is
checked first before doing any manipulations with the destination word.

Beside the tools sources you can find more documentation in file scale_it.c which
calls the renderers.

-------------------------------------------------------------------------------
further explanations on the scalers ... (excerpt from a mail)
-------------------------------------------------------------------------------

Well, the auto-generated scalers are a bit tricky. The main idea is to use
hardcoded "btst" parameters (depending on the height to scale to)
to guess if a pixel should be set or cleared.
Therefore no loops or similar are necessary. The scalers can be seen as
"very stupid" unrolled loops. Each scaler gets called twice. One time for
the upper part and one time for the lower part. Suppose we want to scale
a strip to height 4:

For height 4 only 2 pixels will be drawn for the upper part and 2 pixels for
the lower part.
The indices within the 32-bit texture (for the upper part) are: 31 and 15.

In C the scaler would look like this:

void ScaleToHeight4(unsigned long texture,unsigned short* address,
                    unsigned short setmask, unsigned long clrmask)
{
   // test bit 31 in texture ...
   if (texture & 0x80000000) *address |= setmask;  // set pixel in output
   else                      *address &= clrmask;  // clear pixel in output
   address++;  // step to next destination word

   // test bit 15 in texture ...
   if (texture & 0x8000)     *address |= setmask;
   else                      *address &= clrmask;
}

The destination buffer is arranged in 3 large strips each of width 32 bit.
This way the next destination address (the next line) can be reached with
a simple incrementation of a WORD pointer. These content of these strips
is copied afterwards into the real destination buffer. This additionally
copying can be done very fast, so the runtime gain of just using WORD
operations in the scaler will outweight it completely.
